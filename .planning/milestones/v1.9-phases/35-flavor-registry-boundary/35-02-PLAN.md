## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Phase 32/33 TSV evidence -> static Rust metadata | Checked-in planning inputs are copied into typed Rust constants and can drift if copied incorrectly. |
| Static metadata -> developer lookup API | Developers may treat registry metadata as runtime behavior unless names and tests stay planning-oriented. |
| Capability provenance -> base flavor identity | Base capability evidence can cite downstream-observed source rows without converting base Slic3r into a downstream fork. |
| Deferred/cautioned inventory rows -> future implementation scope | Network/cloud/credential/non-free plugin rows can become implementation scope if the registry exposes dispatch or support semantics. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-35-02-02 | Tampering | Source-reference provenance | mitigate | Use typed `VendorSourceRef` canonical constructors and exact inventory IDs from checked-in TSV rows; tests assert provenance for all base rows and representative downstream rows. |
| T-35-02-03 | Repudiation | Ownership/status interpretation | mitigate | Keep `FeatureOrigin` and `ChecklistStatus` as separate typed fields; tests distinguish `NeedsReview` checklist status from absent `UnknownNeedsReview` inventory evidence. |

| T-35-02-06 | Denial of Service | Lookup helpers | accept | Helpers iterate over small static slices only; no unbounded IO, parsing, allocation-heavy work, or external calls are introduced. |
